    📜 코딩 표준

    프로젝트의 일관성, 가독성, 유지보수성을 높이기 위한 코딩 표준을 안내합니다.

    universal 보편적 규칙

    모든 소스 파일(Rust, Python, Cython 등)에 적용되는 규칙입니다.

    • 들여쓰기: 탭(tab) 대신 스페이스(space)만 사용합니다.
    • 줄 길이: 한 줄은 100자를 넘지 않도록 합니다.
    • 철자법: 미국식 영어 철자법을 사용합니다. (예: color, serialize)

    주석 규칙

    • 주석 블록 위에는 항상 한 줄의 공백을 두어 코드와 시각적으로 분리합니다.
    • 문장의 첫 글자는 대문자로 시작합니다.
    • 명확하고 간결하게, 코드만으로 파악하기 어려운 이유를 설명하는 데 집중합니다.

    Docstring 스타일

    • Python: 명령형으로 작성합니다. (예: Return a cached client.)
    • Rust: 서술형으로 작성합니다. (예: Returns a cached client.)

    🐍 Python 스타일 가이드

    PEP-8 준수

    전반적으로 PEP-8 스타일 가이드를 따릅니다. 단, 몇 가지 예외 및 권장 사항이 있습니다.

    • None 확인: if not foo: 대신 if foo is None: 사용을 권장합니다. 이는 Cython에서 더 효율적인 코드를 생성하고, 예상치 못한 객체로 인한 논리적 오류를 방지합니다.
    • 빈 컬렉션 확인: 리스트나 딕셔너리가 비어있는지 확인할 때는 if not my_list:와 같이 Python의 truthiness를 활용하는 것이 좋습니다.

    타입 힌트

    모든 함수와 메서드 시그니처에는 반드시 포괄적인 타입 어노테이션을 포함해야 합니다.

    def on_bar(self, bar: Bar) -> None:
def on_save(self) -> dict[str, bytes]:

    Docstrings

    문서 자동 생성을 위해 NumPy docstring 사양을 일관되게 준수해야 합니다.

    테스트 메서드 이름

    테스트하는 시나리오를 명확하게 설명하는 서술적인 이름을 사용합니다.

    def test_sma_with_no_inputs_returns_zero_count(self):

    ⚙️ 도구 및 커밋 메시지

    Ruff

    코드 린팅(linting) 도구로 ruff를 사용합니다. 관련 규칙은 프로젝트 최상단의 pyproject.toml 파일에 정의되어 있습니다.

    커밋 메시지

    • 제목: 60자 이하, 대문자로 시작, 마침표 없이 작성합니다.
    • 내용: "무엇을 할 것인지"를 설명하는 명령형으로 작성합니다. (예: Add feature for X, Fix bug in Y)
    • 본문 (선택 사항): 제목과 한 줄 띄고, 변경 사항에 대한 상세한 설명을 추가할 수 있습니다.
